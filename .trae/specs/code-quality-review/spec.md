# 代码质量审查规范

## Why

项目经过多轮迭代开发，需要从代码实现层面进行质量审查，识别潜在的代码问题、性能瓶颈、安全隐患和可维护性问题，以提升整体代码质量。

## What Changes

- 审查代码实现质量
- 检查编码规范一致性
- 识别潜在bug和安全隐患
- 评估性能优化空间
- 检查类型安全

## Impact

- Affected specs: 整体代码质量
- Affected code: 所有源代码文件

---

## 一、代码质量审查维度

### 1.1 代码风格与规范

- TypeScript 类型使用是否完整
- 命名规范是否一致
- 代码格式化是否统一
- 注释是否充分且有意义

### 1.2 代码复杂度

- 函数/组件是否过长
- 圈复杂度是否过高
- 是否存在嵌套过深
- 是否存在重复代码

### 1.3 潜在问题

- 是否存在未处理的异常
- 是否存在内存泄漏风险
- 是否存在竞态条件
- 是否存在空值引用风险

### 1.4 性能问题

- 是否存在不必要的重渲染
- 是否存在内存泄漏
- 是否存在大型依赖未优化
- 是否存在阻塞主线程的操作

### 1.5 安全问题

- 是否存在 XSS 风险
- 是否存在敏感信息泄露
- 是否存在不安全的依赖

### 1.6 可维护性

- 代码是否易于理解
- 是否遵循 SOLID 原则
- 是否有足够的测试覆盖

## 二、审查范围

- `src/app/api/` - API端点

### 2.3 配置文件

- `next.config.ts`
- `eslint.config.mjs`
- `tsconfig.json`

## 三、质量评分标准

### 3.1 评分维度（满分100分）

| 维度       | 权重 | 说明                                |
| ---------- | ---- | ----------------------------------- |
| 类型安全   | 20%  | TypeScript 类型覆盖率、any 使用情况 |
| 代码规范   | 15%  | 命名、格式化、注释                  |
| 代码复杂度 | 15%  | 函数长度、圈复杂度、嵌套深度        |
| 错误处理   | 15%  | 异常处理、边界情况                  |
| 性能       | 15%  | 渲染优化、内存管理                  |
| 安全性     | 10%  | XSS、敏感信息、依赖安全             |
| 可测试性   | 10%  | 代码可测试程度                      |

### 3.2 等级划分

| 等级 | 分数范围 | 说明                 |
| ---- | -------- | -------------------- |
| A    | 90-100   | 优秀，代码质量高     |
| B    | 80-89    | 良好，有少量改进空间 |
| C    | 70-79    | 一般，需要改进       |
| D    | 60-69    | 较差，需要重点改进   |
| F    | <60      | 差，需要重构         |

## 四、审查输出

### 4.1 问题分类

| 级别    | 说明                   | 示例                  |
| ------- | ---------------------- | --------------------- |
| 🔴 严重 | 可能导致崩溃或安全问题 | 未处理的异常、XSS漏洞 |
| 🟡 警告 | 影响性能或可维护性     | 过长函数、缺少类型    |
| 🟢 建议 | 代码风格改进           | 命名不规范、缺少注释  |

### 4.2 输出格式

每个问题包含：

- 文件路径和行号
- 问题级别
- 问题描述
- 改进建议
- 示例代码（如适用）
