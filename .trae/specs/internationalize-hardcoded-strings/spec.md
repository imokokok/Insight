# 国际化硬编码字符串规范

## Why
项目中存在大量硬编码的中英文字符串（约 500+ 处），分散在组件、错误消息、图表标签等位置。这导致：
1. 维护困难：修改文本需要在多处查找和修改
2. 多语言支持不完整：部分内容无法切换语言
3. 代码可维护性差：文本与业务逻辑混杂

## What Changes
- 将所有用户可见的硬编码字符串提取到国际化文件中
- 保持现有的 next-intl 架构和 i18n 文件结构
- 扩展 en.json 和 zh-CN.json 文件，添加缺失的翻译 key
- 在组件中使用 useI18n() hook 替换硬编码字符串
- **BREAKING**: 部分组件的 props 接口可能需要调整以支持国际化

## Impact
- Affected specs: 用户界面、错误处理、图表组件
- Affected code: 
  - 所有包含硬编码字符串的 TSX/TS 文件
  - `src/i18n/en.json` 和 `src/i18n/zh-CN.json`
  - `src/lib/i18n/provider.tsx`（可能需要扩展）

## 实施优先级

### 高优先级（用户直接可见）
1. 用户认证模块（登录、注册）
2. 设置模块（个人资料、通知、偏好、数据管理）
3. 告警模块
4. 收藏模块
5. 错误处理和空状态

### 中优先级（功能相关）
6. 图表组件（价格图表、跨预言机比较）
7. 预言机页面（API3、Redstone、UMA）
8. 首页组件

### 低优先级（开发调试）
9. Console 日志消息（建议保留英文）
10. API 错误消息（部分可保留英文）

## 国际化 Key 命名规范

```
模块.子模块.具体项
例如：
- auth.login.title
- settings.profile.saveSuccess
- alerts.condition.above
- chart.tooltip.open
- common.edit
```

## 注意事项

1. **保持一致性**：使用统一的命名规范
2. **避免重复**：通用文本放在 `common` 命名空间下
3. **参数化**：支持动态参数，如 `{symbol}`, `{value}` 等
4. **Console 日志**：开发调试信息可保留英文，不做国际化
5. **测试覆盖**：确保国际化后功能正常，文本正确显示
