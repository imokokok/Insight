# 多预言机对比功能专业审查 Spec

## Why
用户请求对多预言机对比功能进行专业审查，发现存在架构设计不一致、数据管理问题、性能优化空间、用户体验缺陷等多方面问题，需要系统性地记录并提供改进建议。

## What Changes
- 统一数据架构，消除硬编码数据与实时数据的割裂
- 优化性能，减少不必要的重渲染
- 改进用户体验，增强错误处理和加载状态
- 提升代码质量，减少重复代码
- 增强功能完整性

## Impact
- Affected specs: 市场概览功能、跨预言机对比功能、预言机详情页
- Affected code:
  - `src/app/[locale]/home-components/OracleMarketOverview.tsx`
  - `src/app/[locale]/home-components/OraclePrefetchCard.tsx`
  - `src/components/oracle/charts/CrossOracleComparison/` 目录下所有文件
  - `src/lib/oracles/` 相关客户端

---

## 发现的问题详情

### 问题 1: 数据架构不一致（高优先级 - 架构问题）

**位置**: 
- `OracleMarketOverview.tsx` 第 67-552 行
- `CrossOracleComparison/index.tsx` 第 52-113 行

**描述**:
- `OracleMarketOverview` 使用完全硬编码的静态数据（marketShareData, tvsTrendDataByRange）
- `CrossOracleComparison` 使用实时 API 获取数据
- 两个组件展示相同类型的预言机对比信息，但数据来源完全不同
- 用户在不同页面看到的"市场份额"数据可能不一致

**影响**: 
- 数据可信度降低
- 维护成本增加（需要同时维护两套数据）
- 用户困惑（不同页面数据不一致）

**建议方案**:
1. 创建统一的数据服务层 `OracleMarketDataService`
2. `OracleMarketOverview` 应该从 API 获取实时数据
3. 实现数据缓存策略，避免频繁请求
4. 添加数据过期时间显示

---

### 问题 2: 时间范围选择器数据不真实（高优先级）

**位置**: `OracleMarketOverview.tsx` 第 80-552 行

**描述**:
- `tvsTrendDataByRange` 包含 7 种时间范围的数据
- 每种时间范围的数据都是独立硬编码的
- 数据之间没有逻辑关联（例如 1H 数据的最后一点应该接近 24H 数据的第一点）
- 数据变化过于平滑和线性，不符合真实市场波动

**影响**:
- 用户看到的趋势图是虚假的
- 无法反映真实的市场变化
- 可能误导用户决策

**建议方案**:
1. 接入真实的历史数据 API
2. 实现数据聚合逻辑（如 1H 数据从分钟级数据聚合）
3. 添加"数据来源"说明
4. 如果暂时无法获取真实数据，添加"模拟数据"标签

---

### 问题 3: 缺乏错误边界和错误处理 UI（高优先级）

**位置**: 
- `CrossOracleComparison/index.tsx` 第 52-113 行
- `OraclePrefetchCard.tsx` 第 81-102 行

**描述**:
- `CrossOracleComparison` 中如果某个预言机 API 失败，只记录日志，用户看不到错误信息
- 没有错误边界组件包裹
- 部分预言机失败时，用户不知道哪些数据不可用
- `OraclePrefetchCard` 的预取失败时没有 UI 反馈

**影响**:
- 用户不知道数据是否完整
- 调试困难
- 用户体验差

**建议方案**:
1. 添加 `OracleErrorBoundary` 组件
2. 在 UI 中显示失败的预言机列表
3. 提供重试按钮
4. 显示数据完整性指标（如 "7/10 预言机数据可用"）

---

### 问题 4: 性能优化不足（中优先级）

**位置**: 
- `OracleMarketOverview.tsx` 第 713-865 行
- `CrossOracleComparison/useComparisonStats.ts` 全文件

**描述**:
- `renderTable` 函数在每次渲染时重新创建
- `useComparisonStats` 中的 `heatmapData` 使用 `Math.random()` 生成，导致每次渲染都不同
- 没有使用虚拟化列表处理长列表（如 30+ 个交易对）
- 图表组件没有懒加载

**影响**:
- 不必要的重渲染
- 性能下降
- 内存占用增加

**建议方案**:
1. 使用 `useMemo` 缓存 `heatmapData`，移除随机数生成
2. 对长列表使用 `react-window` 或 `react-virtualized`
3. 使用 `React.lazy` 懒加载图表组件
4. 优化 `renderTable` 函数，拆分为独立组件

---

### 问题 5: 缺乏数据缓存策略（中优先级）

**位置**: `CrossOracleComparison/index.tsx` 第 52-113 行

**描述**:
- 每次切换交易对或预言机选择时，都会重新请求所有数据
- 没有利用 React Query 的缓存能力
- 自动刷新时会丢弃之前的历史数据
- 没有实现请求去重

**影响**:
- 不必要的网络请求
- 服务器压力增加
- 用户等待时间增加

**建议方案**:
1. 使用 React Query 的 `useQuery` 替代手动 fetch
2. 配置合理的 `staleTime` 和 `cacheTime`
3. 实现请求去重
4. 保留历史数据，只追加新数据点

---

### 问题 6: 一致性评分算法过于简单（中优先级）

**位置**: `useComparisonStats.ts` 第 129-141 行

**描述**:
- 当前算法仅基于价格变异系数 (CV)
- 公式 `score = 100 - cv * 10` 过于简化
- 没有考虑：
  - 预言机的历史可靠性
  - 数据源的可信度
  - 更新频率的一致性
  - 极端市场条件下的表现

**影响**:
- 评分不能真实反映预言机质量
- 可能误导用户

**建议方案**:
1. 引入多维度评分模型：
   - 价格一致性 (30%)
   - 更新频率一致性 (20%)
   - 历史可靠性 (25%)
   - 数据源可信度 (15%)
   - 响应时间稳定性 (10%)
2. 添加评分算法说明文档
3. 允许用户自定义权重

---

### 问题 7: 缺乏数据导出功能对比（中优先级）

**位置**: `OracleMarketOverview.tsx`

**描述**:
- `CrossOracleComparison` 有完整的导出功能
- `OracleMarketOverview` 没有任何导出功能
- 用户无法保存市场概览数据

**影响**:
- 功能不完整
- 用户无法进行离线分析

**建议方案**:
1. 在 `OracleMarketOverview` 添加导出按钮
2. 支持 CSV、JSON 格式导出
3. 支持图表截图导出

---

### 问题 8: 预言机选择限制不合理（低优先级）

**位置**: 
- `CrossOracleComparison/index.tsx` 第 152-166 行
- `SettingsTab.tsx` 第 111-137 行

**描述**:
- 限制最多选择 5 个预言机
- 但某些场景下用户可能需要比较更多
- 限制逻辑硬编码在组件中
- 没有解释为什么限制为 5 个

**影响**:
- 限制了用户的使用场景
- 缺乏灵活性

**建议方案**:
1. 将限制提升到配置文件
2. 添加"高级模式"允许选择更多
3. 在 UI 中解释限制原因（如性能考虑）
4. 考虑使用颜色区分而非限制数量

### 问题 9: 缺乏预言机元数据展示（低优先级）

**位置**: `OraclePrefetchCard.tsx`

**描述**:
- 卡片只显示名称、TVS、市场份额
- 缺乏重要信息：
  - 更新频率

  - 最近更新时间

**影响**:
- 用户需要进入详情页才能获取更多信息
- 信息密度低

**建议方案**:
1. 添加悬停时显示更多详情
2. 使用图标表示关键指标
3. 添加健康状态指示器

2. 移动端使用简化视图
3. 表格支持横向滚动
4. 图表字体和间距适配

## 总结

当前多预言机对比功能的基础架构是合理的，但存在以下主要问题：

1. **数据可信度**: 硬编码数据降低了功能的可信度
2. **架构一致性**: 两个主要组件的数据源不一致
3. **用户体验**: 缺乏完善的错误处理和加载状态
4. **性能**: 存在不必要的重渲染和缺乏缓存策略
5. **功能完整性**: 缺乏导出、高级分析等功能

建议按照优先级逐步改进，首先解决数据架构问题，然后优化用户体验和性能。
