# 价格查询功能专业审查 Spec

## Why

用户请求对价格查询功能进行全面的专业审查，提供改进建议以提升用户体验、性能和代码质量。

## What Changes

本审查将从架构设计、用户体验、性能优化、数据质量、错误处理、可访问性等多个维度提供专业建议。

## Impact

- Affected code:
  - `src/app/[locale]/price-query/` 整个目录
  - `src/lib/oracles/` 预言机客户端
  - 相关 hooks 和工具函数

---

## 专业审查报告

### 一、架构设计审查

#### 1.1 状态管理问题 (HIGH)

**现状分析**:
`usePriceQuery` hook 返回了 **70+ 个状态值和方法**，这表明 hook 承担了过多职责，违反了单一职责原则。

**问题**:

```typescript
export interface UsePriceQueryReturn {
  // 70+ 个属性和方法...
}
```

**专业建议**:

1. **拆分 Hook**: 将 `usePriceQuery` 拆分为多个专注的 hooks:
   - `usePriceQueryState` - 管理查询状态
   - `usePriceQueryData` - 管理数据获取
   - `usePriceQueryChart` - 管理图表相关逻辑
   - `usePriceQueryExport` - 管理导出功能
   - `usePriceQueryHistory` - 管理历史记录

2. **使用 Context 模式**: 对于跨组件共享的状态，考虑使用 React Context 替代 prop drilling

3. **状态归组**: 将相关状态组合成对象:

   ```typescript
   // 当前
   const [selectedOracles, setSelectedOracles] = useState(...)
   const [selectedChains, setSelectedChains] = useState(...)
   const [selectedSymbol, setSelectedSymbol] = useState(...)

   // 建议
   const [queryConfig, setQueryConfig] = useState({
     oracles: [],
     chains: [],
     symbol: 'BTC',
     timeRange: 24
   })
   ```

#### 1.2 数据流设计 (MEDIUM)

**现状分析**:
数据获取逻辑与 UI 组件紧密耦合，缺乏清晰的分层。

**专业建议**:

1. **引入 Service 层**: 创建 `PriceQueryService` 处理业务逻辑
2. **使用 React Query / SWR**: 替代手动 fetch 管理，获得:
   - 自动缓存
   - 后台刷新
   - 请求去重
   - 离线支持

### 二、性能优化审查

#### 2.1 请求优化 (HIGH)

**现状分析**:

```typescript
for (const provider of currentSelectedOracles) {
  for (const chain of currentSelectedChains) {
    // 顺序执行每个请求
    const price = await client.getPrice(...)
    const history = await client.getHistoricalPrices(...)
  }
}
```

**问题**:

- 请求是顺序执行的，N 个 oracle × M 个 chain = N×M 次串行请求
- 没有请求缓存，每次切换都会重新获取
- 没有请求优先级

**专业建议**:

1. **并行请求**: 使用 `Promise.all` 或 `Promise.allSettled` 并行获取数据

   ```typescript
   const promises = selectedOracles.flatMap((provider) =>
     selectedChains.map((chain) => fetchPriceData(provider, chain))
   );
   const results = await Promise.allSettled(promises);
   ```

2. **实现请求缓存**:
   - 短期缓存 (30s-60s) 用于实时价格
   - 长期缓存 (5-10min) 用于历史数据
   - 使用 `Map` 或 `lru-cache` 实现

3. **请求优先级**:
   - 当前可见的图表数据优先加载
   - 后台预加载用户可能查看的数据

#### 2.2 渲染优化 (MEDIUM)

**现状分析**:
`PriceChart` 组件在每次数据变化时都会重新计算所有数据。

**专业建议**:

1. **虚拟化长列表**: 如果数据点很多，考虑使用虚拟滚动
2. **图表数据采样**: 对于大数据集，实现动态采样
3. **Web Worker**: 将数据处理逻辑移到 Web Worker

#### 2.3 内存优化 (LOW)

**现状分析**:
历史数据存储在内存中，没有清理机制。

**专业建议**:

1. 实现数据过期清理
2. 限制内存中的历史数据量
3. 使用 `WeakMap` 存储临时计算结果

### 三、用户体验审查

#### 3.1 加载状态 (HIGH)

**现状分析**:
加载时只显示进度条，用户无法看到之前的数据。

**专业建议**:

1. **骨架屏**: 使用骨架屏替代简单的 loading 提示
2. **渐进式加载**: 先显示缓存数据，然后更新
3. **乐观更新**: 用户操作后立即反馈，后台同步
4. **部分加载**: 每获取到一个数据源就立即显示

#### 3.2 错误处理 (HIGH)

**现状分析**:

```typescript
catch (error) {
  logger.error(`Error fetching ${provider} on ${chain}`, error)
  // 错误只记录到日志，用户看不到
}
```

**问题**:

- 错误信息对用户不可见
- 没有重试机制
- 部分失败时用户体验差

**专业建议**:

1. **错误可视化**: 在 UI 中明确显示哪些数据源失败
2. **智能重试**:
   - 指数退避重试
   - 网络恢复后自动重试
   - 用户手动重试单个数据源
3. **降级策略**: 部分失败时显示可用数据
4. **错误分类**: 区分网络错误、数据错误、权限错误

#### 3.3 交互设计 (MEDIUM)

**现状分析**:

- 没有撤销/重做功能
- 没有查询预设
- 没有数据对比的直观展示

**专业建议**:

1. **快捷键支持**:
   - `Cmd/Ctrl + K` 快速搜索
   - `Cmd/Ctrl + R` 刷新
   - `Cmd/Ctrl + E` 导出

2. **查询预设**: 允许用户保存常用查询配置

3. **数据对比增强**:
   - 并排对比视图
   - 差异高亮
   - 统计显著性测试

### 四、数据质量审查

#### 4.1 数据验证 (HIGH)

**现状分析**:
获取的价格数据没有进行有效性验证。

**专业建议**:

1. **价格合理性检查**:
   - 价格不能为负数
   - 价格变化不能超过合理阈值 (如 1 小时内变化超过 50% 应警告)
   - 检测异常值

2. **时间戳验证**:
   - 检查数据是否过期
   - 检测时间戳是否在未来
   - 检测时间序列是否连续

3. **数据源可信度**:
   - 实现数据源评分系统
   - 多源交叉验证
   - 异常数据标记

#### 4.2 数据一致性 (MEDIUM)

**现状分析**:
不同预言机返回的数据格式可能不一致。

**专业建议**:

1. **数据标准化层**: 统一不同数据源的格式
2. **缺失数据处理策略**:
   - 前值填充 (已实现)
   - 线性插值
   - 明确标记缺失区域

### 五、可访问性审查

#### 5.1 键盘导航 (MEDIUM)

**现状分析**:
部分交互元素不支持键盘操作。

**专业建议**:

1. 所有交互元素可通过 Tab 访问
2. 图表支持键盘导航
3. 添加 skip links

#### 5.2 屏幕阅读器支持 (MEDIUM)

**现状分析**:
图表数据对屏幕阅读器不友好。

**专业建议**:

1. 为图表添加数据表格视图
2. 使用 `aria-label` 描述数据变化
3. 提供语音播报功能

#### 5.3 颜色对比度 (LOW)

**现状分析**:
部分颜色对比度可能不足。

**专业建议**:

1. 确保所有文本对比度 >= 4.5:1
2. 不仅依赖颜色传达信息 (添加形状/图案)
3. 支持高对比度模式

### 六、安全性审查

#### 6.1 输入验证 (MEDIUM)

**现状分析**:
用户输入的 symbol 没有严格验证。

**专业建议**:

1. 白名单验证 symbol
2. 防止 XSS 攻击
3. 限制输入长度

#### 6.2 敏感数据处理 (LOW)

**现状分析**:
导出功能可能包含敏感信息。

**专业建议**:

1. 导出前确认
2. 敏感数据脱敏选项
3. 导出日志记录

### 七、可维护性审查

#### 7.1 代码组织 (MEDIUM)

**现状分析**:

- 文件较大 (usePriceQuery.ts 有 964 行)
- 组件职责不够清晰

**专业建议**:

1. 按功能拆分文件
2. 提取通用逻辑到 utils
3. 使用 barrel exports 简化导入

#### 7.2 类型安全 (LOW)

**现状分析**:
部分类型定义不够严格。

**专业建议**:

1. 避免使用 `any`
2. 使用 discriminated unions
3. 实现 runtime type checking (zod)

#### 7.3 测试覆盖 (HIGH)

**现状分析**:
缺乏单元测试和集成测试。

**专业建议**:

1. 关键业务逻辑单元测试
2. 组件快照测试
3. E2E 测试覆盖主要流程
4. 测试覆盖率目标: 80%+
